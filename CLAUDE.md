# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

This is a single self-contained static HTML file with no build, test, or lint tooling and no dependencies. There is no `package.json`.

- **Run/preview**: open `prototype-mock.html` directly in a browser (double-click, or `open prototype-mock.html` on macOS).
- **No build step**: all HTML, CSS, and JS live inline in the one file. Edit it directly and reload the browser to see changes.

## Architecture

`prototype-mock.html` renders a clickable device-frame prototype (an iPhone/iPad mock) centered on a black stage, driven by a single in-memory `state` object (`device`, `orientation`, `lang`, `step`, `apiKey`) inside an IIFE at the bottom of the file. There is no framework, router, or persistence (including the API key — it lives only in memory and is lost on reload) — everything is plain DOM manipulation and re-rendered via one `render()` function whenever state changes.

Key structural pieces, top to bottom in the file:

- **`.topbar` controls**: steps-menu toggle (icon-only, hamburger), settings, the API key control, rotate, and restart — each wired to a click handler (or built dynamically) that mutates `state` and calls `render()`. Device and language are no longer top-level buttons; they live inside the settings popover. The hamburger and the nav arrows are the only icon-only controls by design — new topbar buttons should default to a text label like `settingsBtn`/`orientationBtn`/`restartBtn`, not a `.btn-icon` glyph.
- **`.device` / `.device-screen`**: the visual phone/tablet frame. Switching `state.device` toggles the `iphone`/`ipad` class on `#device`; both classes only ever set base (portrait) width/height/radius/padding — no JS-side layout logic beyond the class swap.
- **Orientation (`state.orientation`, `.landscape` class)**: rotation is done by literally rotating the whole `#device` element 90deg counter-clockwise (`transform: rotate(-90deg)`) rather than swapping its width/height. This makes the frame, notch, and home indicator rotate together as one rigid body with zero extra CSS (a notch that's top-center in portrait ends up correctly on a side in landscape purely from the rotation, no orientation-specific notch/home-indicator rules needed). The one thing that must NOT visually rotate is the actual UI content (step badge/label), so that content lives in `.screen-overlay` — a **sibling** of `.device-screen` (not a descendant, so it isn't clipped by `.device-screen`'s `overflow: hidden`), absolutely positioned and centered on `#device`, with explicit pixel dimensions per device (`.device.iphone .screen-overlay` / `.device.ipad .screen-overlay`, matching the device's inner screen size) that swap in landscape (`.device.iphone.landscape .screen-overlay` etc.) plus a counter-rotation (`rotate(90deg)` — opposite sign from `.device.landscape`, since it must cancel that rotation out). Since both rotations are centered on the same point, the -90/+90 cancel out and the overlay's content renders upright at the correct swapped dimensions while the frame around it is rotated. If you need to add more "screen content" (vs. decorative frame chrome like the notch), it belongs inside `.screen-overlay`, not `.device-screen`. The device's rotate/resize transitions and the screen-overlay's counter-rotation transition share the same duration and `cubic-bezier(0.4, 0, 0.2, 1)` easing so they animate in lockstep — keep them matched if you touch either.
- **`.screen-content`**: placeholder per-step content (currently just a step badge/number and label) rendered inside `.screen-overlay`. Extend this if real per-step mockup content is added.
- **Nav arrows (`#prevBtn`/`#nextBtn`)**: live outside the device entirely, as direct children of `#stage` flanking it left/right (not inside `.screen-overlay`), so they never rotate with the device and stay upright regardless of orientation. Their `left`/`top` are set in JS by `positionNavArrows()`, which computes the device's *target* on-screen box analytically from `DEVICE_SIZES[state.device]` + `state.orientation` (swapping width/height when landscape) rather than from `deviceEl.getBoundingClientRect()`. This matters: `render()` changes `#device`'s class and then immediately calls `positionNavArrows()` in the same tick, while `#device`'s rotation is a CSS transition — reading its live rect at that instant returns the *pre*-transition box (the transition hasn't advanced yet), which made the arrows visibly lag one click behind. Computing from state instead of the DOM sidesteps the race entirely. `stageEl.getBoundingClientRect()` is still read live (safe — `#stage` itself never animates) to find the viewport center the device is aligned to. `positionNavArrows()` runs on every `render()` and on `window resize`. The CSS transition on `.nav-arrow`'s `left`/`top` (same duration/easing as the device) is what makes them glide smoothly to their new spot instead of jumping. Both arrows and `.bottombar` progress dots call the shared `goToStep(n)`, which clamps to `1..TOTAL_STEPS` and re-renders.
- **Popovers (`.menu-overlay` / `.menu-panel`)**: a shared pattern used by three anchored popovers — the steps menu (`#menuOverlay`), settings (`#settingsOverlay`), and the API key entry popup (`#apiKeyOverlay`). Each is a transparent full-screen click-catcher with a small anchored panel inside; clicking outside the panel (or `Escape`) closes it. `openMenu()`/`openSettings()` close each other so only one top-level popover is open at a time; `closeSettings()` also closes the nested API key popover.
- **Settings popover**: built by `buildSettingsPanel()`, rebuilt on every `render()`. Contains a language segmented control (`#langSegmented`, EN/DE/FR) and a device segmented control (`#deviceSegmented`, iPhone/iPad).
- **API key control (`#apiKeyRow`)**: lives in the topbar next to the Settings button (not inside the settings popover), built by `buildApiKeyButton()` on every `render()`. Shows either an "Add key" button (no key set) or a masked chip (`maskKey()` — fixed 8 dots + last 4 characters) with a remove (✕) button.
- **API key entry popover (`#apiKeyOverlay`)**: opened via `openApiKeyPopover(anchorEl)`, which positions `#apiKeyPanel` under the triggering button using `getBoundingClientRect()` (not a fixed offset like the other popovers, since the anchor button's position varies). `Enter` in the input or the Save button commits via `saveApiKey()`; empty input is ignored.
- **i18n**: the `i18n` object holds `en`/`de`/`fr` string tables (including a `step(n)` function for the numbered label). `t()` returns the active language's table. `applyStaticI18n()` generically walks every `[data-i18n]` element and sets `textContent` from the matching key (and `[data-i18n-placeholder]` for input placeholders) — add a new string by giving the element a `data-i18n="key"` attribute and adding that key to all three language tables, no manual wiring needed in `render()`. Dynamically-built elements (segmented control buttons, the API key add/remove button) are labeled directly from `t()` in their build functions instead, since they don't exist in the static markup.

## Conventions to preserve when editing

- `TOTAL_STEPS` is the single source of truth for step count — dots, menu items, and nav-arrow disabled states all derive from it. Change step count there, not by editing markup.
- `LANGS` (`["en", "de", "fr"]`) is the single source of truth for available languages — the language segmented control is built from it. Adding a language means adding it to `LANGS` and adding a matching key to `i18n` with the same shape as the existing entries.
- Adding a static UI string: give the element `data-i18n="key"` in markup and add that key (as a string, not a function) to **all** language tables — `applyStaticI18n()` picks it up automatically. Strings needed only in JS (e.g. dynamically-built buttons) are read via `t().key` directly in the relevant build function.
- Device-frame visual differences (notch vs. camera dot, corner radius, bezel padding) are expressed purely in CSS via the `.iphone`/`.ipad` classes on `#device` — don't branch on `state.device` in JS beyond the class swap. Orientation is a rotation (`transform: rotate(-90deg)` on `.device.landscape`, counter-clockwise), not a dimension swap — don't reintroduce explicit landscape width/height/notch-position overrides on `.device-screen`; any new frame-level decoration should just work automatically since it rotates with `#device`. If you ever flip the rotation direction again, the `.screen-overlay` landscape rules must flip their `rotate()` sign to match (they must always be the exact negation of `.device.landscape`'s angle to cancel out to upright).
- Anything the user should perceive as on-screen "app content" (not frame chrome) must go inside `.screen-overlay`, sized via the `.device.<type> .screen-overlay` / `.device.<type>.landscape .screen-overlay` rules, so it stays upright when rotated. Don't add new absolutely-positioned content directly to `.device-screen` — it will rotate (and be clipped) with the frame instead of staying upright.
- The nav arrows are positioned in JS from `DEVICE_SIZES`/`state`, not by reading `deviceEl.getBoundingClientRect()` — the device is mid-transition when `render()` runs, so live geometry reads its stale pre-transition box. Keep `DEVICE_SIZES` in sync with the `.device.iphone`/`.device.ipad` width/height in CSS, and do the same "compute from state, don't read a transitioning element's rect" for any future controls that need to track the device.
- Disabled affordances (e.g. Restart on step 1, nav arrows at the ends) use the native `disabled` attribute / a `.disabled` class plus `pointer-events: none`, set from `render()` — keep new disabled states consistent with this pattern rather than removing elements from the DOM.
- New popovers should follow the existing `.menu-overlay`/`.menu-panel` pattern (transparent click-catcher + anchored panel, open/close toggled via a class, outside-click and `Escape` dismiss) rather than introducing a different modal style.
